# API

Base URL for local development:

```text
http://localhost:8000
```

## GET /health

Purpose: check backend availability.

Response:

```json
{
  "status": "ok",
  "app": "SmartBuyAgent",
  "version": "0.1.0"
}
```

## POST /api/chat

Purpose: run a non-streaming shopping-guide or knowledge-answer request.

Request:

```json
{
  "query": "预算3000，推荐一款拍照好的手机",
  "session_id": null,
  "debug": true
}
```

Response:

```json
{
  "answer": "...",
  "product_cards": [
    {
      "product_id": "phone_001",
      "title": "...",
      "brand": "...",
      "price": 2999,
      "image_url": "https://example.com/images/phone_001.jpg",
      "tags": ["拍照好", "性价比"],
      "attributes": {
        "存储容量": "256GB"
      },
      "source_url": "https://example.com/products/phone_001",
      "compare_url": "https://example.com/compare/phone_001",
      "recommend_reason": "..."
    }
  ],
  "citations": [
    {
      "chunk_id": "doc_phone_phone_camera_guide_chunk_001",
      "title": "手机拍照选购指南",
      "section": "...",
      "section_path": "...",
      "source_file": "data/knowledge_docs/phone/phone_camera_guide.md",
      "content_preview": "...",
      "score": 0.82
    }
  ],
  "trace": [
    {
      "step": "query_understanding",
      "intent": "shopping_guide"
    }
  ],
  "session_id": "generated-or-reused-session-id"
}
```

Notes:

- `query` is required and cannot be blank.
- `session_id` is optional. If omitted, the backend creates a new session ID.
- `debug=false` returns an empty `trace` list.
- Product cards and citations are not generated by the LLM.
- `query_understanding` trace steps include the unified structured result. When rule confidence is low, the backend may call an LLM only for JSON slot extraction; `llm_fallback_attempted`, `llm_fallback_status`, `source`, `confidence`, and `reason` describe that path.
- The returned `answer` is validated by AnswerGroundingGuard before it is accepted as final text. The trace includes `answer_grounding_guard` with pass/fallback status and violation summaries.
- The endpoint can return `429 Too Many Requests` when the same session exceeds the short request window.

## POST /api/chat/stream

Purpose: run the same chat workflow and return Server-Sent Events for debug display.

Request:

```json
{
  "query": "预算3000，推荐一款拍照好的手机",
  "session_id": null,
  "debug": true
}
```

Response content type:

```text
text/event-stream
```

Events:

```text
event: session
data: {"session_id":"..."}

event: node_start
data: {"node":"product_retrieval","status":"running","started_at":"..."}

event: retrieval
data: {"type":"product","returned_products":3,"cache_status":"miss"}

event: token
data: {"node":"response_compose","delta":"根据你的预算，"}

event: answer_draft_delta
data: {"node":"response_compose","delta":"Draft LLM wording for debug display only"}

event: grounding_guard_result
data: {"status":"fallback","violations":[{"type":"purchase_boundary_violation"}]}

event: final_answer
data: {"answer":"Guarded final answer shown in the chat message","status":"fallback"}

event: stream_guard
data: {"node":"response_compose","status":"blocked","reason":"purchase_action"}

event: node_end
data: {"node":"product_retrieval","status":"success","duration_ms":83}

event: trace
data: {"step":"query_understanding","intent":"shopping_guide"}

event: result
data: {"answer":"...","product_cards":[],"citations":[],"trace":[],"session_id":"..."}

event: done
data: {"status":"ok"}
```

On stream-level failure:

```text
event: error
data: {"failed_node":"knowledge_retrieval","error_type":"RuntimeError","message":"..."}

event: done
data: {"status":"error"}
```

Notes:

- Stage Stream-2 emits realtime node-level events while the workflow is running.
- `node_end.duration_ms` reports per-node elapsed time.
- Product and knowledge retrieval are exposed as separate stream nodes, so the frontend can show their independent start/end timing.
- Retrieval events summarize product recall, knowledge retrieval, or in-session comparison without sending long source text.
- Token events prefer provider-native LLM streaming through `stream_chat`; providers without native streaming fall back to chunked non-streaming output. Product cards and citations still come only from retrieval/comparison services.
- `stream_guard` is emitted when streaming answer text matches high-risk phrases such as purchase actions, fabricated purchase links, skincare treatment/drug-effect claims, or fabricated source language.
- If a stream is guarded, token output is interrupted and the final `result.answer` is replaced with a safe fallback. The final `result` event remains authoritative; frontend clients should overwrite any partial token text with `result.answer`.
- `answer_draft_delta` is draft/debug text only. It is not the official chat answer. Frontend clients should show the official message from `final_answer` and then reconcile with the final `result.answer`.
- `grounding_guard_result` reports whether the final answer passed AnswerGroundingGuard or fell back because of unsupported price, purchase-boundary, unsupported fact, brand/product, citation-support, or skincare-safety violations.
- Memory saving is handled by the API layer and stores the original query.
- Query understanding in stream mode uses the same structured result as `/api/chat`; trace events expose whether the LLM JSON-slot fallback was skipped, succeeded, or failed.
- The endpoint can return `429 Too Many Requests` before an SSE stream starts.

## POST /api/feedback

Purpose: store user feedback for an answer.

Request:

```json
{
  "session_id": "session-id",
  "turn_id": null,
  "rating": "helpful",
  "reason": "recommendation_relevant",
  "comment": "The answer was useful.",
  "query": "预算3000，推荐一款拍照好的手机",
  "answer_preview": "Short answer preview..."
}
```

Response:

```json
{
  "id": 1,
  "status": "saved"
}
```

Notes:

- `rating` must be `helpful` or `not_helpful`.
- `comment` is limited to 1000 characters.
- `answer_preview` is truncated server-side to 500 characters.
- Feedback is stored for evaluation and does not affect current ranking.
- Redis feedback counters are best-effort short-term aggregates; MySQL/SQLite remains the persistent feedback source.
